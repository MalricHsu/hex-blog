# 技術使用指南（為什麼用 × 怎麼用）

這份文件針對本專案用到的每個技術，說明**為什麼要用它（解決什麼問題）**、**怎麼用（步驟與多種用法）**、以及**注意事項**，方便作為使用方法的紀錄與複習。

---

## 1. 為什麼用 Nuxt（而不是純 Vue）

### 為什麼

純 Vue（CLI / Vite）做出來的是 **SPA（單頁應用）**，會遇到幾個痛點：

1. **SEO 差**：SPA 初始 HTML 幾乎是空的，靠 JS 才把畫面畫出來，搜尋引擎/社群分享抓不到內容。
2. **首屏慢**：使用者要等 JS 下載執行完才看到畫面。
3. **沒有後端**：要另外開一個 Node 專案做 API。
4. **很多東西要自己接**：路由、SEO meta、狀態管理都要手動設定。

Nuxt 解決這些：

- **SSR（伺服器端渲染）**：在伺服器先把 HTML 連同資料產生好再丟給瀏覽器 → SEO 好、首屏快。
- **檔案路由**：放檔案就有路由，不用寫 router 設定。
- **內建後端（Nitro）**：`server/` 直接寫 API，前後端同一個專案。
- **自動匯入**：元件、composable、store 都自動匯入，少寫一堆 import。

### 一句話

> Nuxt = Vue + SSR + 檔案路由 + 內建後端 + 自動匯入，是「全端框架」。

## 2. 檔案路由（File-based Routing）

### 為什麼

傳統要手寫一份 router 設定表（哪個網址對哪個元件），網站一大就難維護。Nuxt 改用「**檔案結構即路由**」，一看資料夾就知道有哪些頁。

### 怎麼用

| 檔案 | 網址 | 說明 |
| --- | --- | --- |
| `pages/index.vue` | `/` | 首頁 |
| `pages/about.vue` | `/about` | 一般頁 |
| `pages/blog/index.vue` | `/blog` | 資料夾首頁 |
| `pages/blog/[id].vue` | `/blog/1` | 動態路由（單一參數） |
| `pages/[...slug].vue` | 任意路徑 | catch-all（萬用） |

**取得動態參數：**

```js
const route = useRoute();
console.log(route.params.id);   // 網址 /blog/1 → "1"
console.log(route.path);        // "/blog/1"
console.log(route.query.page);  // 網址 ?page=2 → "2"
```

**頁面間導覽（用 NuxtLink，不要用 a）：**

```vue
<NuxtLink to="/blog">部落格</NuxtLink>
<NuxtLink :to="`/blog/${blog.id}`">看文章</NuxtLink>  <!-- 動態要加 : -->
```

> `NuxtLink` 是前端路由跳轉（不重整整頁），比 `<a>` 快。

### 注意

- 動態網址一定用 `:to`（綁變數），純字串用 `to`。

## 3. Layouts（版型）

### 為什麼

每頁都有 Nav、Footer，如果每頁重寫一遍，改一個地方要改很多檔。Layout 把「共用外框」抽出來，頁面只負責中間的內容。

### 怎麼用

```vue
<!-- app/layouts/default.vue -->
<template>
  <Nav />
  <slot />     <!-- 頁面內容會塞進這個位置 -->
  <Footer />
</template>
```

```vue
<!-- app/app.vue -->
<template>
  <NuxtLayout>      <!-- 套用 layout -->
    <NuxtPage />    <!-- 目前頁面 -->
  </NuxtLayout>
</template>
```

**某頁要用不同版型：**

```vue
<!-- pages/contact.vue -->
<script setup>
definePageMeta({ layout: "no-footer" });  // 用 layouts/no-footer.vue
</script>
```

### 補充

- 沒指定 layout 的頁，自動用 `default`。
- 想完全不用 layout：`definePageMeta({ layout: false })`。
- `<slot />` 是 Vue 的插槽，意思是「把外層包進來的內容放這裡」。

## 4. 自動匯入（Auto-imports）

### 為什麼

少寫 import，程式更乾淨。Nuxt 在背後維護一份「哪些東西可以直接用」的索引。

### 怎麼用（哪些會自動匯入）

| 放在哪 | 自動匯入到 | 範例 |
| --- | --- | --- |
| `app/components/` | 全部頁面/元件 | `<FormAlert />` |
| Vue / Nuxt API | 全部 | `ref`、`computed`、`useFetch`、`useRoute` |
| `app/composables/` | 前端 | 自己寫的 `useXxx()` |
| `app/stores/`（複數！） | 前端 | `useToastStore()` |
| `server/utils/` | 只有伺服器 | API 內用的工具 |
| `shared/utils/` | 前端 + 伺服器 | 共用 schema |

### 注意（踩雷）

- `stores` 要**複數**才會自動匯入，放成 `store`（單數）就要手動 import。
- `server/utils/` 的東西**在前端元件拿不到**（只有伺服器端）。前後端都要用 → 放 `shared/utils/`。
- 新增這些資料夾後**要重啟 dev server**，索引才會更新。

## 5. 資料取得：useFetch / useAsyncData

### 為什麼

純 Vue 抓資料要這樣寫：

```js
const data = ref([]);
onMounted(async () => {
  const res = await fetch("/api/x");
  data.value = await res.json();
});
```

問題：只在瀏覽器抓（SSR 抓不到、SEO 差）、要自己管 loading/error。

Nuxt 的 `useFetch` 幫你：**SSR 階段就在伺服器抓好**、回傳響應式資料、自動處理 loading/error，所以**不用 `onMounted`**。

### 怎麼用

```js
// 最簡單
const { data: service } = await useFetch("/api/service");

// 完整回傳
const { data, pending, error, refresh } = await useFetch("/api/service");
// data    = 資料（ref）
// pending = 是否載入中
// error   = 錯誤
// refresh = 手動重抓的函式
```

**常用參數：**

```js
const { data } = await useFetch("/api/blogs", {
  default: () => [],          // 資料還沒來時的預設值（避免 null）
  lazy: true,                 // 不卡住頁面切換，背景載入
  transform: (res) => res.items,  // 先處理資料再回傳
});
```

**自訂抓取邏輯用 `useAsyncData`：**

```js
// 第一個參數是「唯一 key」（快取用），第二個是抓取函式
const { data: blogs } = await useAsyncData("blog-list", () =>
  queryCollection("blog").all()
);
```

### 什麼時候還是要 onMounted

`onMounted` 是給「**一定要在瀏覽器、DOM 準備好才能做的事**」：

```js
onMounted(() => {
  new Swiper(".swiper", { ... });   // 香草版套件初始化
  // 讀 window/document 尺寸、addEventListener 等
});
```

### 注意

- `useFetch`/`useAsyncData` 要 `await`（在 `<script setup>` 最外層可用 top-level await）。
- `data` 是 ref，`<script>` 內存取要 `.value`，模板自動解包。

## 6. 伺服器 API（Server Routes）

### 為什麼

有些事不能在前端做：

- **保護機密**（API key、資料庫密碼不能進前端）。
- **驗證不能只信前端**（前端可被繞過）。
- **整合第三方**（寄信、金流）。

Nuxt 內建 Nitro，`server/` 直接寫後端，不用另開專案。

### 怎麼用

```js
// server/api/send-contact.post.js  （.post = 只接受 POST）
export default defineEventHandler(async (event) => {
  const body = await readBody(event);          // 讀前端送來的資料
  const result = contactSchema.safeParse(body); // 後端再驗一次
  if (!result.success) {
    throw createError({ statusCode: 400, message: "資料錯誤" });
  }
  // ……寄信 / 存 DB……
  return { success: true, message: "已送出" };  // 回傳給前端
});
```

**前端呼叫：**

```js
const res = await axios.post("/api/send-contact", formData);
console.log(res.data.message);  // "已送出"
```

### 補充

- 檔名決定方法：`.get.js`、`.post.js`、`.put.js`、`.delete.js`。
- 動態路由：`server/api/blog/[id].get.js` → 用 `getRouterParam(event, "id")` 取參數。
- 取得 query：`getQuery(event)`。
- `readBody`、`createError`、`getQuery` 都是 Nitro 自動匯入。

## 7. shared 目錄（前後端共用）

### 為什麼

表單驗證要做兩次：**前端驗**（即時提示使用者）+ **後端驗**（防繞過）。如果兩邊各寫一份規則，容易不一致。把規則寫一份放 `shared/`，兩邊共用。

### 怎麼用

```js
// shared/utils/schema.js
import { z } from "zod";
export const contactSchema = z.object({ /* 規則 */ });
```

- 前端元件：直接用 `contactSchema`（自動匯入）。
- 伺服器 API：也直接用 `contactSchema`（自動匯入）。

### 注意

- 只能放「**前後端都能跑的純邏輯**」（zod、純函式、常數）。
- 不能放只有伺服器有的東西（`readBody`、`process.env`、`fs`）→ 那種留 `server/utils/`。

## 8. Nuxt Content（用 Markdown 寫部落格）

### 為什麼

文章內容如果寫死在 JS 陣列裡，又長又難維護、不好排版。Nuxt Content 讓你**像寫筆記一樣用 `.md` 檔寫文章**，支援標題、表格、圖片，還能自動產生路由。

> 適用「自己寫稿、git 管理」的部落格；若要「後台讓使用者線上新增」，則要改用資料庫。

### 怎麼用

**1. 定義 collection（`content.config.ts`）：**

```ts
export default defineContentConfig({
  collections: {
    blog: defineCollection({
      type: "page",            // page = 對應網址
      source: "blog/**/*.md",
      schema: z.object({       // frontmatter 欄位型別
        image: z.string(),
        categories: z.array(z.string()),
        date: z.string(),
      }),
    }),
  },
});
```

**2. 寫文章（`content/blog/1.md`，檔名即網址 `/blog/1`）：**

```md
---
title: 文章標題
categories: [UIUX 設計]
date: "2025-10-16"
---

## 內文
正文……
```

**3. 撈資料（各種查法）：**

```js
queryCollection("blog").all();                       // 全部
queryCollection("blog").path("/blog/1").first();     // 單篇（依路徑）
queryCollection("blog").where("categories", "LIKE", "%UIUX%").all();  // 條件
queryCollection("blog").order("date", "DESC").limit(3).all();         // 排序+取前 3
```

**4. 渲染內文：**

```vue
<ContentRenderer v-if="blog" :value="blog" />
```

### 注意

- 元件用 `queryCollection("blog")`；伺服器 route 要 `queryCollection(event, "blog")`（簽名不同）。
- 渲染出的 HTML 樣式要用 `:deep()` 才套得到。

## 9. 頁面標題與 SEO（useHead / useSeoMeta）

### 為什麼

每頁瀏覽器分頁標題、Google 搜尋結果、社群分享預覽都要能各自設定，對 SEO 與使用者體驗很重要。

### 怎麼用

```js
// 只設標題
useHead({ title: "服務項目" });

// 設 SEO（標題 + 描述 + 社群預覽）
useSeoMeta({
  title: "服務項目",
  description: "提供品牌設計、網頁設計與前端開發服務。",
  ogTitle: "服務項目 | Nelson Blog",
  ogImage: "https://.../og.png",
});
```

**全域樣板（`nuxt.config.ts`）：**

```ts
app: {
  head: {
    htmlAttrs: { lang: "zh-Hant" },      // 語言（SEO/無障礙必備）
    titleTemplate: "%s | Nelson Blog",   // %s 代入各頁 title
    title: "Nelson Blog",                 // 沒設 title 的頁用這個
    link: [{ rel: "icon", type: "image/svg+xml", href: "/favicon.svg" }],
  },
},
```

### useSeoMeta 參數說明

每個參數會被轉成對應的 `<meta>` 標籤，出現在不同地方：

| 參數 | 產生的標籤 | 作用 / 顯示在哪 |
| --- | --- | --- |
| `title` | `<title>` + `<meta property="og:title">` | 分頁標題、搜尋結果標題、分享卡片標題 |
| `description` | `<meta name="description">` | Google 搜尋結果標題底下的摘要 |
| `ogType` | `<meta property="og:type">` | 內容類型：`website`（一般頁）/ `article`（文章） |
| `ogSiteName` | `<meta property="og:site_name">` | 分享卡片上的網站名稱 |
| `ogImage` | `<meta property="og:image">` | 分享到 FB / LINE 的預覽縮圖（建議 1200×630） |
| `twitterCard` | `<meta name="twitter:card">` | Twitter/X 卡片樣式：`summary_large_image`（大圖）/ `summary`（小圖） |
| `ogUrl` | `<meta property="og:url">` | 這頁的標準網址 |
| `ogLocale` | `<meta property="og:locale">` | 語系，如 `zh_TW` |
| `robots` | `<meta name="robots">` | 是否讓搜尋引擎收錄，如 `index, follow` |

**什麼是 OG（Open Graph）？** `og:` 開頭的是 Facebook 制定、現在 LINE / Discord / Slack 通用的標準。貼連結到這些平台時，它們會去抓 `og:*` 標籤組成「分享預覽卡片」（縮圖 + 標題 + 描述）。沒有 `og:*`，貼連結就只是一條乾巴巴的網址。

> `useSeoMeta` 的好處：你設 `title` / `description`，它會**自動補上對應的 `og:title` / `og:description`**，不用手動寫兩份。

### 本專案的完整做法

**① 全域預設（`app.vue`）** — 沒被各頁覆蓋就用這份：

```js
useSeoMeta({
  description: "Nelson 的設計與前端作品集……",
  ogType: "website",
  ogSiteName: "Nelson Blog",
  ogImage: "https://.../og.png",
  twitterCard: "summary_large_image",
});
```

**② 各頁覆蓋** — 每頁寫專屬 title + description：

```js
useSeoMeta({
  title: "服務項目",
  description: "提供品牌識別、網頁設計、前端切版與後端開發等服務。",
});
```

**③ 動態頁（文章詳情）** — 用箭頭函式讓它響應式：

```js
useSeoMeta({
  title: () => blog.value?.title,
  description: () => blog.value?.description || blog.value?.title,
  ogType: "article",
  ogImage: () => blog.value?.image,   // 用文章封面當分享圖
});
```

### SEO 檢查清單

| 項目 | 為什麼 |
| --- | --- |
| `<html lang>` | 告訴搜尋引擎/螢幕閱讀器語言 |
| 每頁 `title` | 分頁標題、搜尋結果標題 |
| 每頁 `description` | 搜尋結果摘要 |
| `og:*`（標題/描述/圖） | 社群分享預覽卡片 |
| 動態頁 SEO | 每篇文章各自的 meta |
| sitemap.xml | 讓搜尋引擎收錄全站（可裝 `@nuxtjs/sitemap`） |

### 注意

- 動態頁要用**箭頭函式** `() => blog.value?.title`，資料回來才會更新（SSR 也正確）。
- 首頁記得給自己的 title，否則套用預設會變「Nelson Blog | Nelson Blog」。
- 分享預覽要部署到**正式網域**才抓得到（localhost 抓不到）；可用 [opengraph.xyz](https://www.opengraph.xyz/) 測試。

## 10. Pinia（跨元件共用狀態）

### 為什麼

有些狀態要**很多元件共用**（登入狀態、全域通知）。如果用 props 一層一層往下傳，會變成「props 地獄」。Pinia 提供一個「全域倉庫」，任何元件都能直接讀寫。

> 頁面內自己用的狀態（表單欄位、開關）用 `ref` 就好，不用 Pinia。

### 怎麼用

**1. 安裝：** `npm i pinia @pinia/nuxt`，在 `nuxt.config` 的 `modules` 加 `@pinia/nuxt`。

**2. 建 store（`app/stores/toast.js`）：**

```js
export const useToastStore = defineStore("toast", () => {
  const toasts = ref([]);                       // state
  const success = (msg) => {                     // action
    toasts.value.push({ id: Date.now(), type: "success", msg });
  };
  return { toasts, success };                    // 要用的才 return
});
```

**3. 使用：**

```js
const toast = useToastStore();
toast.success("成功！");      // 呼叫 action（不是 toast.success = "...")
console.log(toast.toasts);    // 讀 state
```

### 補充：兩種寫法

```js
// setup 寫法（像 script setup，本專案用這個）
defineStore("id", () => { const x = ref(0); return { x }; });

// options 寫法（像 Vue Options API）
defineStore("id", {
  state: () => ({ x: 0 }),
  actions: { inc() { this.x++; } },
});
```

### 注意

- state 用 `ref`；action 是函式要**呼叫**（`toast.success("...")`，不是 `=`）。
- store 放 `stores`（複數）才自動匯入。

## 11. Zod（資料驗證）

### 為什麼

手寫一堆 `if (!name) ... if (!email.includes("@")) ...` 又長又容易漏。Zod 用「宣告式 schema」一次定義所有規則，還能回傳清楚的錯誤訊息，前後端共用。

### 怎麼用

```js
import { z } from "zod";

const contactSchema = z.object({
  name: z.string().min(2, "請輸入姓名"),
  phone: z.string().regex(/^09\d{8}$/, "手機格式不正確"),
  email: z.string().email("email 格式不正確"),
  age: z.number().min(18).optional(),   // 選填
});
```

**驗證（用 safeParse，不會丟錯）：**

```js
const result = contactSchema.safeParse(formData);
if (!result.success) {
  // 取出所有錯誤訊息
  const msgs = result.error.issues.map((i) => i.message).join("、");
  console.log(msgs);
} else {
  console.log(result.data);   // 驗證通過、型別正確的資料
}
```

### 補充：常用規則

```js
z.string().min(2).max(20)
z.string().email()
z.string().url()
z.string().regex(/.../)
z.number().min(0).max(100)
z.boolean()
z.array(z.string())
z.enum(["a", "b"])
z.object({ ... })
.optional()        // 可不填
.default("值")     // 預設值
```

### 注意

- `safeParse` 回傳 `{ success, data | error }`，不會丟例外；`parse` 失敗會丟錯。
- 錯誤在 `result.error.issues`（陣列，每個有 `.message`）。

## 12. Bootstrap 客製 + scoped 樣式

### 為什麼

Bootstrap 提供現成 utility class（`d-flex`、`col-md-6`、`text-center`…），切版快。客製 SCSS 變數可以把品牌色、字級接進這套系統，沿用 Bootstrap 的機制（自動生成 `bg-primary-900`、按鈕 hover…）。

### 怎麼用

**客製主題色（`_variables.scss`）：**

```scss
$theme-colors: (
  "primary-900": #24140b,
  "neutral-0": #ffffff,
);
// → 自動可用 bg-primary-900、text-neutral-0、btn-primary-900…
```

**scoped 樣式 + `:deep()`：**

```vue
<style scoped>
/* 只作用在這個元件 */
.box { color: red; }

/* 要穿透到「子元件根」或「動態產生的內容」(如 ContentRenderer) */
:deep(table) { width: 100%; }
</style>
```

### 注意

- `<style scoped>` 預設碰不到子元件內部與動態 HTML，要 `:deep()`。
- 紅色波浪底線若是 cSpell/ESLint，跟 Prettier 無關（Prettier 只排版）。

## 13. RWD 響應式策略

### 為什麼

同一個網站要在手機、平板、桌機都好看。Bootstrap 用「斷點」(sm/md/lg/xl) 控制不同寬度的版型。

### 怎麼用（本專案的策略）

**1. 「補 base class」**：`xxx-md-OO` 只在該斷點以上生效，手機要補沒斷點的 base：

```
py-md-20  →  py-10 py-md-20    （手機 py-10、桌機 py-20）
```

**2. 斷點選擇**：本專案把結構斷點設在 `lg`(992)，讓平板也用手機堆疊版型（`col-lg-*`、`navbar-expand-lg`），避免平板擠到跑版。

**3. scoped 內用 media query**（inline style 不能寫 media query）：

```css
@media (max-width: 991.98px) { .x { ... } }   /* 平板以下 */
@media (min-width: 992px)    { .x { ... } }   /* 桌機以上 */
```

**4. 顯示/隱藏切換**：

```html
<div class="d-none d-lg-block">桌機才顯示</div>
<div class="d-lg-none">手機/平板才顯示</div>
```

### 注意

- 用 `991.98px` 而非 `992`，避免和 `min-width: 992px` 在整數點重疊。

## 14. 開發流程小結

一個功能從無到有，大致流程：

1. **規劃資料來源**：寫死假資料（`server/utils`）／API（`server/api`）／Markdown（`content/`）。
2. **建頁面**：在 `pages/` 開檔，`useFetch`/`useAsyncData` 取資料。
3. **切版**：Bootstrap utility + 客製 SCSS，邊做邊補 RWD base class。
4. **互動邏輯**：`ref`/`computed` 管狀態，事件處理；跨元件共用就上 Pinia。
5. **表單**：`shared/` 寫 zod schema，前端先驗 → 送 `server/api` → 後端再驗 → 回傳訊息（toast 顯示）。
6. **抽元件**：重複 UI 抽成 `components/`。
7. **SEO**：每頁 `useHead`。
8. **測試 + 微調**：RWD、hover、邊界情況。

### 何時重啟 dev server

- 改 `nuxt.config.ts`、裝套件、加 module。
- 新增 `shared/`、`stores/`、`layouts/`、`content.config.ts`。
- （一般 template/script/style 改動會熱更新，不用重啟。）
